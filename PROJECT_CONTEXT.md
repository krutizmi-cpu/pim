# PROJECT_CONTEXT.md

Единый файл контекста проекта PIM для быстрого входа в работу (человека или агента).
Дата актуализации: 2026-04-28.

## 1) Что это за проект

PIM-система для контент-отдела, где:
- Ozon — эталон структуры категорий и атрибутов;
- мастер-карточка товара накапливает знания из импорта, поставщика, шаблонов клиентов;
- итог — быстрый экспорт в клиентские Excel-шаблоны с минимальным ручным трудом.

Ключевая цель: не заполнять характеристики заново каждый раз, а переиспользовать уже накопленные данные.

## 2) Неподвижные правила (не нарушать)

1. Ozon-категории/атрибуты приоритетнее legacy-категорий из прайсов.
2. Один товар = одна мастер-карточка, которая обогащается со временем.
3. Источник значения отслеживается и учитывается по приоритету.
4. Экспорт всегда должен поддерживать клиентские единицы измерения (конвертацию).
5. Никаких “разовых” решений: повторяемые сценарии сохраняются в профили.

См. также: [PIM_METHOD.md](PIM_METHOD.md).

## 3) Текущий функциональный статус

### Уже реализовано

- Streamlit-приложение с разделами:
  - `Импорт`
  - `Каталог`
  - `Карточка`
  - `Атрибуты`
  - `Клиентский шаблон`
  - `Ozon`
  - `Каналы`
- Импорт Excel (авто/ручной режим лист+строка заголовка), batch-учёт загрузок.
- Каталог с пагинацией, фильтрами, массовыми действиями.
- Supplier enrichment (массовый и точечный):
  - supplier page parsing;
  - fallback-поиск;
  - fallback габаритов из статистики и defaults;
  - source tracking и source priority.
- Ozon sync:
  - кэш дерева категорий;
  - кэш атрибутов и справочников;
  - импорт Ozon-атрибутов в master.
- Клиентские шаблоны:
  - авто-матчинг + ручной матчинг;
  - сохранение mapping rules;
  - сохранение template profiles;
  - preview/gap/readiness;
  - экспорт в структуре шаблона.
- Конвертации единиц и media transforms в template flow.
- UX-улучшения по разделам `Карточка` и `Атрибуты`.
- AI-интеграция (первая рабочая версия):
  - централизованные AI-настройки в отдельном разделе `Настройки` (OpenAI/OpenRouter/NVIDIA);
  - генерация SEO-описания из карточки товара;
  - AI-подсказки для пустых Ozon-атрибутов;
  - генерация 2 маркетинг-изображений по промптам из карточки.
- Медиа-слой:
  - поддержка публичного `base URL` для фото, если в прайсах приходят локальные пути;
  - ZIP-выгрузка фото по товару, текущей странице и всей фильтрации.
- Документы ФСА:
  - поиск деклараций и сертификатов в карточке товара;
  - поиск по юрлицу поставщика, виду товара и `ТН ВЭД`;
  - сохранение реквизитов документа и PDF в историю товара.

### Важные недавние изменения

- В корень проекта добавлен execution-plan [ROADMAP_5_DAYS_1000_SKU.md](ROADMAP_5_DAYS_1000_SKU.md):
  - это жёсткий 5-дневный roadmap на переход к conveyor-подходу;
  - цель зафиксирована как `1000 SKU/day` силами `1 менеджера`;
  - основной акцент: root-domain parser, AI verification/rewrite, 3–5 фото, readiness queues и client-first export.
- `Каталог` начал превращаться в operational queue screen:
  - на текущей странице уже считаются очереди `Нужен parser-run / Не найден релевантный товар / Нет фото / Готово к AI/клиенту`;
  - фокус менеджера начинает строиться вокруг queue-based работы, а не вокруг простого листинга товаров.
- В `Каталог` сервисные действия больше не собраны в один шумный слой:
  - блок `Проверка парсинга и сервис` разделён на вкладки `Ozon категории / Supplier-Web parser / Логистика / Фото и сервис`;
  - это важный контур для быстрой QA-проверки parser-flow без AI.
- Чистый parser-flow теперь диагностируется прозрачнее:
  - после `Только supplier/web enrichment` менеджер видит последние причины ошибок прямо в интерфейсе;
  - у выбранного товара в фокус-блоке показывается текст причины, если парсинг завершился `error`.
- Починен внутренний runtime-баг parser-flow:
  - auto-fill Ozon-атрибутов после supplier enrichment больше не падает на чтении `sqlite3.Row`;
  - это был один из сценариев, из-за которых казалось, что parser “вообще не работает”.

- В `Карточка` рабочий порядок перестроен под реальный сценарий менеджера:
  - сначала поиск и выбор товара;
  - затем hero-шапка `Карточка товара` и метрики по уже выбранной карточке;
  - это убирает ощущение, что пользователь сначала попадает в карточку "вообще", а уже потом должен искать товар.
- Иконки разделов приведены к более спокойному монохромному стилю:
  - вместо разноцветных emoji у разделов используются более аккуратные символы;
  - это лучше сочетается с текущим light-shell интерфейсом и меньше похоже на черновую админку.
- Главная навигация PIM вынесена в горизонтальный верхний ряд:
  - блок разделов больше не забирает место в sidebar;
  - sidebar теперь используется под текущий раздел, память и операционный контекст.
- Ускорен single-item AI flow в `Карточка`:
  - сокращены token caps для названия, описания и AI-подсказок атрибутов;
  - уменьшен NVIDIA chat-timeout, чтобы простые операции не “висели” по несколько минут.
- Усилен supplier parser на сценариях, где в товаре сохранён root/listing URL поставщика:
  - такие страницы теперь сильнее штрафуются по relevance;
  - parser реже принимает главную страницу сайта за реальную product card.
- В `Карточка` кнопка `Заполнить пайплайн` переименована в более понятную `Заполнить карточку автоматически`.
- Явный блок поиска/выбора товара в `Карточка`.
- Улучшенный UX `Атрибуты` (инструкция, фильтры, вкладки, экспорт списка).
- Видимость профилей шаблонов (`Текущий канал`/`Все каналы`).
- Стабилизация постоянной БД:
  - по умолчанию PIM должен жить на `~/.pim/catalog.db`, а не на случайной временной базе;
  - если обнаружены несколько `catalog.db`, система должна не только выбрать базу, но и попытаться слить в `~/.pim/catalog.db` данные из известных копий;
  - это нужно, чтобы товары, Ozon-атрибуты, Ozon-каталог, mapping rules, template profiles и AI/system settings не “пропадали” после нового входа.
- Постоянная память по загруженным файлам:
  - импортные каталоги поставщиков должны сохраняться в `~/.pim/uploads/imports/`;
  - клиентские шаблоны должны сохраняться в `~/.pim/uploads/templates/`;
  - в БД должны накапливаться `uploaded_files` и `catalog_import_history`, чтобы было видно, что уже загружалось и откуда взялась партия.
- Backup-память как отдельный слой:
  - критичные изменения данных должны периодически фиксироваться в `~/.pim/backups/db/`, а не только жить в текущей SQLite;
  - snapshot кэша Ozon должен сохраняться не только на скачивание, но и как серверная копия в `~/.pim/backups/ozon/`;
  - менеджер должен иметь возможность восстановить Ozon snapshot прямо из памяти сервиса, без повторной ручной загрузки файла.
- Реестр клиентов и повторное использование шаблонов:
  - менеджер не должен вручную вспоминать и вводить код клиента каждый раз;
  - список клиентов должен браться из БД, с возможностью создать нового клиента прямо в UI;
  - сценарий `Клиентский шаблон` должен позволять открыть уже сохранённый Excel по связке `клиент + категория` без повторной загрузки файла.
- Sportmaster category templates as persistent client memory:
  - Excel-шаблоны Sportmaster нужно хранить не как разовый upload, а как category-specific overlay memory по `Атрибутный класс id`;
  - при импорте шаблона в память должны сохраняться сам `.xlsx`, category code `sportmaster:<attr_class_id>`, required flags, instructions, справочные значения и стартовые mapping rules;
  - повторяющиеся колонки шаблона (`Вид спорта*`, `Вид спорта*.1`, `Вид спорта*.2`) нельзя схлопывать при регистрации профиля и правил;
  - Sportmaster rules нужно держать рядом с шаблоном: цена обязательна для продажи, UPC передаётся как `0 + UPC`, фото идут прямыми публичными ссылками, а после товародвижения нельзя свободно править штрихкод и ВГХ.
- Detmir как клиентский overlay, а не второй canonical core:
  - правило проекта сохраняется: `Ozon = эталон категорий и атрибутов`;
  - `Детский Мир` подключается как клиент с собственными category requirements и карточками, но не как новый master-источник товаров;
  - для Detmir теперь есть read-only API-sync в `Каналы`: дерево категорий, категории с атрибутами, значения справочников и кэш карточек товаров;
  - Detmir schema и content-card snapshots должны сохраняться в постоянной БД и переживать рестарт/деплой так же, как Ozon cache и клиентские шаблоны;
  - импорт Detmir category requirements в PIM должен идти в overlay-слой (`channel_attribute_requirements`, `channel_mapping_rules`) без смены Ozon-ядра.
  - рабочий flow менеджера должен идти через `Карточка`: выбрать Detmir-категорию, импортировать её требования, заполнить gaps из master/Ozon-данных и увидеть readiness/payload до отправки.
- AI-провайдеры:
  - базовые AI-настройки по-прежнему хранятся в `system_settings`;
  - дополнительно нужны именованные `ai_connection_profiles`, потому что у NVIDIA/OpenRouter/OpenAI-compatible сценариев токены и модели могут часто меняться.
  - UX-правило: общие AI-настройки не должны жить в `Каналы`; для них нужен отдельный системный раздел `Настройки`.
  - рабочее правило: загрузка AI-профиля должна уметь сразу делать его активной конфигурацией, а не просто наполнять форму.
- AI-first enrichment:
  - основной практический сценарий должен смещаться в сторону `Ozon category -> supplier/web -> AI`;
  - в `Каталог` нужен массовый AI-пакет по текущей странице и по всей фильтрации;
  - AI должен уметь не только писать SEO-описание, но и собирать продающее название товара без лишней “маркетинговой пены”.
  - UI должен показывать этот сценарий как основной, а не прятать его среди десятка одноуровневых кнопок.
- UX выбора товара:
  - в `Карточка` ключевой идентификатор — артикул;
  - выбор товара должен быть удобен через быстрый список и сортировку по совпадению артикула, а не только через длинный selectbox.
- Shell/UI рабочего пространства:
  - основной вход в PIM должен выглядеть как единое рабочее пространство, а не набор равнозначных вкладок;
  - навигация, состояние памяти и текущая БД лучше читаются через sidebar + hero shell;
  - визуально приоритет должен быть у надёжной light data-workspace подачи, а не у “прототипных” экранов Streamlit.
- UX `Каталога` и `Карточки`:
  - каталог должен быть ближе к “операционной админке”: быстрый список по артикулу, понятный этап карточки, статус парсинга, наличие фото и быстрый переход к редактированию;
  - карточка должна начинаться со сводки состояния товара и не перегружать пользователя длинной однослойной формой;
  - редактирование лучше группировать по смысловым блокам: идентификация, категории/Ozon, логистика, фото/описание.
  - массовые действия должны уметь работать по shortlist выбранных товаров, а не только по всей странице или всей фильтрации;
  - готовность под Ozon и клиентские шаблоны должна быть видна ближе к месту работы, а не только в отдельных технических блоках.
- Клиентский export flow:
  - сценарий `выбрал профиль -> выбрал пачку товаров -> увидел readiness -> выгрузил` должен быть first-class потоком;
  - shortlist, собранный в `Каталог`, должен без ручного перебора переиспользоваться в `Клиентский шаблон`.
- Ручная корректировка категорий поверх Ozon:
  - `category / base_category / subcategory` должны редактироваться вручную в карточке без привязки только к выпадающим спискам;
  - manual-category не должна перетираться обычным `Ozon` auto-match без `force`;
  - массовое редактирование категорий в `Каталог` должно работать и для Ozon-привязанных товаров.
  - если Ozon-match явно неверный, должна быть явная кнопка очистки Ozon-привязки с последующим повторным матчингом.
- Sportmaster category templates:
  - шаблоны вида `Инвентарь_...` нужно читать не с первого листа, а с `Шаблон для заполнения`, если он есть;
  - для них шапка находится в строке 3, строка 4 содержит маркер `Обязательный атрибут`, а реальные данные начинаются с 5-й строки;
  - при сохранении исходной структуры Excel нельзя писать значения в строку маркеров/обязательности.
- Настройки парсинга в UI + автопайплайн fallback:
  - supplier URL -> Ozon fallback -> Яндекс fallback -> web fallback.
  - глобальные parser/media/AI-настройки должны быть вынесены из рабочих экранов в отдельный раздел `Настройки`;
  - в самом `Каталог` оставлять только параметры текущего запуска и массовые действия по товарам.
- Защита от неверного fallback-совпадения:
  - проверка артикула/кода;
  - проверка overlap токенов названия;
  - проверка бренда/score;
  - отклонение нерелевантного кандидата.
- Защита от supplier-доменов с антиботом:
  - если сайт поставщика отдаёт `401/403` или anti-bot HTML, generic web fallback не должен оставаться привязан к этому домену;
  - причина блокировки должна сохраняться в `supplier_parse_comment`.
  - при блокировке домена fallback должен строить усиленные запросы по `артикулу + бренду + названию` и в первую очередь пробовать `Ozon` / `Яндекс Маркет` / общий web.
- Ozon category -> product attributes:
  - после назначения Ozon-категории должны не только создаваться `requirements`, но и материализоваться пустые слоты атрибутов у товара;
  - эти слоты должны быть видны в `Карточка` и `Атрибуты` без ручного создания.
  - если Ozon cache стал полнее после нового sync/snapshot, PIM должен добирать недостающие `requirements`, а не считать категорию уже полностью импортированной только из-за частичного старого импорта.
  - если supplier parsing заполнил основные поля товара раньше, чем появились Ozon-атрибуты категории, карточка должна уметь автоматически дозаполнить Ozon/category блок из уже накопленных полей и master-атрибутов.
- Source tracking атрибутов:
  - ручное сохранение атрибутов в `Карточка` и `Атрибуты` должно писать source как `attr:<code>`, иначе source priority для атрибутов ломается.
  - AI-подсказки атрибутов должны писать source точно так же: `attr:<code>`, а не в “голый” код.
- Фото и клиентские выгрузки:
  - локальные пути вида `\\\\fs03\\...jpg` нельзя отдавать клиенту как есть;
  - в export flow должны попадать публичные `http/https` ссылки;
  - архив фото должен формироваться с именами `ARTICLE_1`, `ARTICLE_2`, ...
  - в карточке нужен быстрый визуальный просмотр текущих фото товара, иначе пользователь не понимает, что реально сохранилось.
- Референсный шаблон клиента `Детский Мир -> Велосипеды`:
  - лист `Товары`, машинная шапка в строке 1, русская расшифровка в строке 2;
  - конфигурация на листе `Справочник`: `categoryId=68`;
  - на 2026-04-27 авто-матчинг уверенно ловит 26 полей из 61;
  - хорошо матчится база: артикулы, название, штрихкод, бренд, упаковка, габариты, вес, описание, ТН ВЭД, фото;
  - дополнительно зашиты явные Detmir-mappings: страна производства, пол, материал, тип велосипеда/беговела, тип колёс, тип тормоза, цвет, комплектация;
  - отдельного внимания дальше требуют клиентские/category-specific поля: возрастные границы, рекомендованный возраст, герои, конструкция велосипеда, материал рамы, складные, тип тормозной системы, тип вилки, гарантия, рост ребёнка, высоты руля/сиденья, диаметры колёс, цены/скидки, сертификат, модель работы, архивный статус.

## 5) Пайплайн парсинга (факт)

Функция: `enrich_product_from_supplier` в `app.py`.

Порядок:
1. Пытается взять карточку по `supplier_url` (включая `?q=` шаблоны).
2. Если данных мало/подозрительно:
   - Ozon fallback (поиск)
   - Яндекс.Маркет fallback (поиск)
   - generic web fallback
3. Если габаритов всё ещё нет или payload сомнительный:
   - fallback из статистики каталога
   - fallback из category defaults
4. Применение значений с учётом source priority + защиты manual.

Настройки парсинга хранятся в `system_settings` и редактируются в UI (`Каталог`):
- `timeout_seconds`
- `max_hops`
- `fallback_max_results`
- `require_article_match`
- `min_name_overlap`
- `min_fallback_score`
- `enable_web_fallback`
- `enable_ozon_fallback`
- `enable_yandex_fallback`
- `enable_stats_fallback`
- `enable_defaults_fallback`

## 6) Почему “пропадают” профили/данные (диагностика)

Частые причины:
1. Другая БД (путь отличается от ожидаемого).
2. Фильтр профилей по каналу (`Текущий канал`) скрывает профили других каналов.
3. Данные действительно не сохранены в `template_profiles`.

Проверка:
- В UI смотреть `Текущая база данных`.
- В `Клиентский шаблон` переключить `Показывать профили` -> `Все каналы`.

## 8) Операционная цель

- PIM делается под быстрый production-flow: `5–10 тыс. карточек в месяц` силами `1 менеджера`.
- Любая новая UX-правка должна оцениваться через этот критерий:
  - меньше скрытых действий;
  - меньше ручных возвратов;
  - понятнее, где массовый parser-flow, а где тонкая ручная доводка.
- Парсинг + fallback + анти-ошибочные проверки.

### Stage C — Ozon эталон (в работе)
- Полное покрытие sync + стабильный импорт в master + UX проверки.

### Stage D — Клиентские шаблоны (сделано в базовом боевом виде)
- Матчинг, профили, gap, readiness, экспорт.

### Stage E — Точность и надёжность (активный текущий приоритет)
- Улучшение precision parsing, снижение ложных совпадений, лучшее покрытие габаритов/веса.

### Stage F — Маркетинговый блок (позже)
- Рерайт описаний, фото/инфографика.

### Stage F — Маркетинговый блок (старт выполнен)
- Подключён AI-контур (настройки + генерация текста + генерация изображений).
- Следующая итерация: улучшить контроль качества, шаблоны промптов по каналам и пакетную генерацию.

## 8) Что делать следующему агенту в первую очередь

1. Проверить, что используется правильная БД (`data/catalog.db` или `PIM_DB_PATH`).
   - если в UI показан временный путь, это почти наверняка не та база;
   - целевой путь по умолчанию: `~/.pim/catalog.db`.
2. Проверить AI-настройки в `Настройки` и тест соединения (provider/model/key/base_url).
3. Проверить профиль клиента в `Клиентский шаблон`:
   - `Показывать профили = Все каналы`.
4. Прогнать 1 реальный supplier batch и оценить:
   - долю success/error;
   - сколько fallback;
   - качество габаритов.
5. Для слабых категорий усилить извлечение атрибутов в `services/supplier_parser.py`.
6. Проверять для спорной Ozon-категории не только `full_path`, но и конкретный `type_id`: один путь Ozon может иметь несколько типов товара с разным числом атрибутов.
7. Проверить карточку ФСА на реальном товаре:
   - поиск кандидатов;
   - разбор документа;
   - сохранение PDF и реквизитов.
8. Для `Детский Мир -> Велосипеды (categoryId=68)` дожать матчи оставшихся category-specific полей и решить, какие из них становятся master, а какие остаются channel-only.
9. После каждого изменения:
   - `python -m py_compile app.py services/*.py`
   - commit + push.
10. Для Detmir API:
   - не тащить Detmir в роль второго PIM-ядра;
   - использовать его как источник клиентской схемы и read-only снимков карточек;
   - если позже появится запись обратно в Detmir API, делать её отдельным слоем publish/export;
   - barcode под Детский Мир считать first-class blocker;
   - фото под Детский Мир считать условно готовыми только если это публичные URL и желательно не меньше 3;
   - справочные Detmir-значения должны проходить через локальный dictionary cache, иначе payload нельзя считать готовым.

## 9) Быстрый smoke-check перед релизом

1. `Импорт`: новая партия реально добавляется/обновляется.
2. `Каталог`: фильтры, пагинация, массовые кнопки работают.
3. `Карточка`: поиск/выбор товара виден, парсинг выполняется.
4. `Атрибуты`: UX понятен, сохранение значений работает.
5. `Клиентский шаблон`: профиль сохраняется и загружается повторно.
6. `Ozon`: категории/атрибуты кэша отображаются.
