# PIM Method

Этот файл — единый проектный ориентир. Любая новая задача и изменение в коде должны проверяться на соответствие этому документу.

## 1) Цель и миссия

- Цель: максимально сократить ручной труд контент-отдела при подготовке карточек товаров для разных клиентов/каналов.
- Миссия: накапливать знания о товаре в одной мастер-карточке и переиспользовать их во всех следующих шаблонах.

## 2) Базовая модель

- `Ozon` — базовый эталон структуры товара.
- `Ozon` также является отдельным клиентом/каналом продаж.
- Товар может создаваться:
- вручную;
- импортом Excel;
- из Ozon-атрибутов/словарей в мастер-слой.

## 3) Главный принцип данных

- Один товар = одна мастер-карточка (`master card`), которая расширяется со временем.
- Новый клиентский шаблон НЕ создаёт параллельный товар, а обогащает существующую мастер-карточку.
- Система обязана показывать:
- что сматчено;
- что не сматчено;
- чего не хватает для выгрузки.

## 4) Обязательный workflow

1. Импорт/создание товара.
2. Обогащение от поставщика (`supplier_url`, парсинг страницы, фото, характеристики, габариты/вес).
3. Загрузка шаблона клиента (Excel/API).
4. Авто-матчинг колонок + показ результата матчинга.
5. Gap-анализ (что не заполнено).
6. Дозаполнение (авто + руками).
7. Экспорт готового шаблона (один товар/выборка/массив).

## 5) Источники и приоритеты

Источник каждого значения должен быть отслеживаемым.

Приоритет источников (от сильного к слабому):

1. `manual`
2. `client_validated`
3. `supplier_page`
4. `ai`
5. `default`

По умолчанию более слабый источник не перезаписывает более сильный.

## 6) Единицы измерения

- Внутри системы храним нормализованные значения.
- На импорт/экспорт применяем конвертацию под требования клиента.
- Базовые сценарии обязательны: `см/мм/м`, `кг/г`, `inch/cm`.

## 7) Медиа и контент

- Поддерживать работу с фото как с мульти-значениями (главное фото, N-е фото, список ссылок).
- Маркетинговый блок (следующий слой после ядра):
- улучшение/перегенерация описания;
- генерация и обработка изображений;
- инфографика (обязательное требование).

## 8) Что делаем сейчас (приоритет)

1. Укрепляем ядро PIM:
- supplier enrichment;
- source tracking + priority policy;
- template matching + gap + readiness;
- reusable template profiles;
- transforms (включая единицы и media).
2. После стабилизации ядра — визуальный проход (UX/UI) без изменения методологии.

## 9) Чего не делать

- Не переписывать проект с нуля, если задачу можно решить точечной доработкой.
- Не добавлять сложную архитектуру "на будущее", если нет прямой пользы для текущего workflow.
- Не уводить фокус в маркетинговые фичи, пока не закрыто ядро данных/экспорта.

## 10) Правило принятия решений

Если есть несколько вариантов реализации, выбираем тот, который:

1. уменьшает ручной труд уже сейчас;
2. усиливает мастер-карточку и переиспользование данных;
3. повышает прозрачность происхождения значений;
4. улучшает готовность товаров к клиентским шаблонам.
